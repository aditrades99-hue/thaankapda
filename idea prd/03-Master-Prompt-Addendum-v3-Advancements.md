## ◈ PART 11 — ADVANCED FEATURE ENHANCEMENTS [NEW]

### 11.1 — Bilingual Mode (English ⇄ Nepali)

```
☐ Add a language toggle to the navbar: "EN | नेपाली"
☐ Use a lightweight i18n approach (JSON dictionaries per locale, or next-intl)
☐ Translate: hero copy, nav links, category names, CTAs, calculator labels,
  chatbot opening message and quick-reply chips, footer text
☐ Persist the chosen language in a cookie (only after cookie-consent "Essential" tier,
  since this is functional, not tracking)
☐ Ensure no layout shift when switching — both languages should fit
  within the same container widths (test longest Nepali strings)
```

### 11.2 — Festival / Seasonal Theme Variants

```
☐ Add a `themeVariant` config (default | dashain | tihar | teej)
☐ Each variant adds a thin accent ribbon near the top of the hero:
  e.g., "🎉 Dashain Special — Extra 10% off bulk orders this week"
☐ Variants may shift accent colors slightly (e.g., brighter gold/red overlay)
  but MUST preserve the base palette tokens and contrast ratios (4.5:1 minimum)
☐ Variant can be set manually via CMS/config, or auto-activated by date range
```

### 11.3 — Inquiry Reference & "Track My Inquiry" [NEW]

```
☐ Every WhatsApp inquiry generated by the calculator or chatbot includes
  a unique reference code, e.g. MTE-20260614-A93F
  (format: MTE-YYYYMMDD-<4 char random>)
☐ Reference code is:
  — appended to the WhatsApp message ("Ref: MTE-20260614-A93F")
  — stored server-side (e.g., Vercel KV / Supabase) with timestamp,
    fabric, quantity, and mode (retail/wholesale)
☐ New page: /track — customer enters their reference code to see status:
  Received → Quoted → Confirmed → Dispatched
  (status updated manually by staff via a simple admin view, out of scope
   for v1 unless separately contracted)
```

### 11.4 — Referral Teaser [NEW]

```
☐ Small banner/component: "Refer a friend, both get 5% off your next order"
☐ Generates a shareable link with a UTM/ref query param
  (e.g., ?ref=<short-code>) for basic attribution
☐ No login/account system required — referral is tracked only via the
  query param passed through to the WhatsApp message
```

### 11.5 — WhatsApp Broadcast Opt-In [NEW]

```
☐ Inline opt-in form (footer or dedicated section):
  "Get fabric drops & festive offers on WhatsApp"
  — Phone number input (Nepal format, +977)
  — Consent checkbox (required, linked to Privacy Policy)
☐ Validate phone format client-side AND server-side
☐ Sanitize with the same sanitizeInput() helper as the calculator
☐ Store opt-ins server-side for the business to use with WhatsApp
  Business Platform broadcast lists (subject to WhatsApp's messaging policies)
```

### 11.6 — Enhanced AI Stylist [NEW]

```
☐ Multi-turn session memory: send recent message history to /api/chat
  (do not persist full conversation server-side beyond the session
  unless the user opts into lead capture)
☐ "Mood board" mode: user describes an occasion (wedding, Teej, daily wear)
  → AI suggests 2–3 fabric pairings, each linking to a relevant category
☐ Optional voice input via Web Speech API (desktop + supported mobile browsers),
  with a visible mic icon and fallback to text input
```

### 11.7 — SEO & Structured Data [NEW]

```
☐ JSON-LD schemas:
  — LocalBusiness (name, address, phone, hours, geo coordinates, rating)
  — Product (for Hot Products section)
  — BreadcrumbList
  — FAQPage (if an FAQ section is added)
☐ Open Graph + Twitter Card meta tags, with a dynamic OG image via @vercel/og
☐ Auto-generated sitemap.xml and robots.txt
☐ hreflang tags if bilingual mode (11.1) is implemented
```

### 11.8 — Analytics & Conversion Tracking [NEW]

```
☐ Google Analytics 4 (consent-gated — loads only after "Accept All"
  or "Manage Preferences > Analytics")
☐ Custom events:
  — whatsapp_click (FAB, navbar CTA, intent modal, calculator, chatbot)
  — intent_modal_open
  — calculator_used (with mode: retail/wholesale)
  — chatbot_lead_captured
☐ Vercel Analytics for Core Web Vitals monitoring (no consent required —
  privacy-friendly, aggregate only)
```

### 11.9 — Dark / Light Mode Toggle [NEW]

```
☐ Default theme: existing dark luxury palette (Part 2.2 of original prompt)
☐ Optional "Daylight Boutique" light theme using --color-silk / --color-ivory
  as base surfaces, with maroon/gold accents preserved
☐ Toggle in navbar; persisted via cookie (Essential tier)
☐ Respects prefers-color-scheme on first visit, user toggle overrides it
```

### 11.10 — Accessibility Enhancements [NEW]

```
☐ "Skip to content" link, visible on keyboard focus
☐ Focus-visible outlines in --color-gold-light on all interactive elements
☐ aria-live="polite" region for new chatbot messages
☐ All decorative 3D/SVG elements marked aria-hidden="true"
☐ Reinforce: every animation respects prefers-reduced-motion (already
  required in v2.0, Part 10 — verify again in QA)
```

## ◈ PART 12 — LEGAL & COMPLIANCE INTEGRATION [NEW]

### 12.1 — Cookie Consent Banner

```
Component: /components/ui/CookieConsent.tsx

Behavior:
  — Appears on first visit, bottom of viewport, glassmorphic style
    matching the existing design system (var(--glass-bg), var(--glass-blur))
  — Three actions: "Accept All" / "Essential Only" / "Manage Preferences"
  — Blocks Google Analytics / Meta Pixel script injection until consent
    includes "Analytics" or "Marketing"
  — Stores the user's choice (e.g., in a first-party cookie) so the
    banner does not reappear on subsequent visits
  — "Manage Preferences" opens a small modal with per-category toggles
    (Essential — always on / Analytics / Marketing)

Copy (from Legal & Licensing Package, Section 5.2):
  "🍪 We use cookies. We use essential cookies to make this site work,
   and — with your permission — analytics cookies to understand how
   visitors use our site. You can change your choice anytime in Cookie Settings."
```

### 12.2 — Legal Pages

```
New routes:
  /legal/terms     → Terms of Service (Legal & Licensing Package §3)
  /legal/privacy   → Privacy Policy (Legal & Licensing Package §4)
  /legal/cookies   → Cookie Policy (Legal & Licensing Package §5)

Each page:
  — Server-rendered, static (RSC), included in sitemap
  — Linked from the footer on every page (see 12.3)
  — Styled consistently with the site's design system, but content-first
    (prioritize readability — body text in --font-body, max-width ~700px)
```

### 12.4 — Data Sourcing Compliance Update [REPLACES original Part 1] [NEW]

The original Part 1 ("Data Ingestion") instructs fetching data directly from
public Instagram/Facebook/Google URLs. **Update this as follows:**

```
☐ Business info (address, phone, hours, rating): may be manually verified
  and entered once via the Google Business Profile listing — this is a
  one-time setup step, not a runtime scrape.
☐ Reels & Posts (Section 4.6 / 4.7C): MUST be fetched server-side via the
  official Meta Graph API using a token from a Meta Developer App owned
  by Maruti Textile Empire (see Legal & Licensing Package §8). Do not
  fetch or parse instagram.com / facebook.com HTML directly at runtime.
☐ Product names & brand tone: may be derived once, manually, from
  reviewing existing captions/posts during the design phase — not via
  an automated scraper running in production.
```
